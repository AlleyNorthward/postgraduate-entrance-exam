# 面向对象和面向过程思想上的碰撞
- 在嵌入式, 工程实践课等跟硬件相关的课程中, 会用纯C编写, 但是我会选择用C++写, 对硬件进行抽象. 而在软件工程课中, 应该用面向对象编写时, 我却选择了纯C编写的开源项目分析.
- 软件工程这门课似乎更多地以应用软件为主, 而系统软件却遭到了大家的"忽视". 应用级软件会有很多用户需求, 需要灵活地动态改变, 面向对象会有更好地收缩拓展空间, 而系统软件相对稳固, 不用频繁地更改, 可是却对运行速度有着要求, 此时面向过程的C是更优的选择
- 试问有多少同学会用`Interface`以及`abstract class`? 又有多少同学会**真正**使用**运行时多态**?
	- 学`Java`的同学可能还好一点, 有`gc`帮你回收资源
	- 学C++的同学除了需要对面向对象有理解外, 还需要对内存管理有一定的认识. (当然, lua本身在内存管理方面做地就非常好, 很值得学习)
- 假如很少使用上述面向对象技巧的话, 那么对于用户需求的**灵活收缩拓展**方面, 使用面向对象跟面向过程几乎**没有任何区别**, 解决起来都是很棘手的.
	- 为什么会这么假设呢? 
		- 因为学生项目的话, 结果几乎是固定的: 展示完整项目即可, 不用考虑用户需求的变化. 
		- 另外如果依赖Ai的话, 本身输入的需求就是固定的, 也没有变化的一说. 当然除非你**带着需求的变化**询问Ai, 但Ai输出的代码本身**不是你写的**, 除了过课设外, 个人能力的提升还是很少的.
	- 为什么会没有区别? 
		- 注意我并不是说面向对象跟面向过程没有任何区别, 我强调的是用户需求变化的编码层面.
		- 面向对象一般是一个类一个文件, 如果将C也这么看待的话, 那么.c文件中的static类似私有方法, .h中的函数声明是公有方法, 抽象属性可以放在抽象的结构体中
		- 而面向对象突出的优点是**运行时多态**, 可以动态改变, 自废一臂的话, 跟面向过程优势就小了很多
