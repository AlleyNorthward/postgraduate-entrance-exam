# 描述
- 映射这个词, 十分通用, 放在这里是什么意思呢?
- 从计算机底层到计算机上层, 本质上一直都在**寻址**
	- 但是为了统筹分析计算机底层的[[202605311817-直接寻址|直接寻址]],[[202605311818-间接寻址|间接寻址]],与数据结构层面的[[202605311648-存储表|存储表]], 查找之间的关系, 这里会抽象出来直接映射与间接映射的概念思想
- 正常情况下, 会依据[[202605291221-地址|地址]], 直接寻找[[202605291233-内存|内存单元]](或者[[202605311621-存储单元|存储单元]]), 这种情况可以称为**直接映射**, 也就是地址唯一地对应一个内存单元
- 除了直接映射外, 还有**间接映射**, 也就是地址直接映射另一个地址, 这叫作间接映射, 然后呢, 被映射的这个地址会去映射内存单元(或者存储单元)
- 这里还是借鉴线性代数, 引入**定义域**和**值域**的概念, 方便管理统筹, 但是没有那么多的限制.同时引出**键**和**值**的概念
- 对于直接映射和间接映射, 一定是单射, 不一定是满射(比如数组, 有下标, 但是没有值)
# 分析
- 对于直接映射, 每一个值总会有一个唯一的键. 在这种情况下, 十分明确, 不复杂, 例如数组就是这么干的, 下标是键, 数组内部表项是值
	- 需要注意的是, 这个值, 可以存储着**很多信息**
- 对于间接映射, 直观地看, 通过键找到值的同时, 这个值又可以作为键, 寻找另一个值. 看起来十分累赘, 直接找不就行了, 为什么还要过度一下?
	- 其实十分有必要
	- 首先一点是可以**减少首个键的存储空间**, 例如[[202605291018-GDTR寄存器|GDTR寄存器]]这里的分析, GDTR寄存器, 只是一个寄存器, 无法创立多个映射项. 注意, 它是一个48位的寄存器
		- 不过它作为**键**, 可以存储其映射值的信息
		- 当然, 如果只是存储一个地址, 也只能找到一个**值**, 这个值也翻不起什么波浪
		- 可是如果它存储的是[[202605142235-线性存储空间|线性表]]的首地址和偏移, 那么这个**值**, 就是一个数组了, 空间可以根据其最大偏移获取, 也就是$2^{13}$, 共用`8192`个表项!
		- 本质还是一一对应的, 但是并不像数组那样的强制映射, 而是硬件层面提供的软性映射
	- 其次我们可以将关键信息放入值域中的数组的[[202605311621-存储单元|存储单元]]中, 同时, 将它作为键, 指向另外的地方
	- 这样, 我们就能扩大查找的范围了
# 关联
- 所以分析时, 键, 只是用来查找的信息, 有了键, 相当于我们有了值
- [[202605291018-GDTR寄存器|GDT表]]
- [[202605291645-打开分页机制|page目录表, page表]]
	- 这里就有多级映射的概念, 首先, 有个分页相关的键, 指向了gdt表项, 偏移+gdt表项中又作为键, 可以找到page目录表表项, 根据其表项的分析为, 可以判断直接通过MMU分析还是再次作为键, 找到page表, 再通过MMU分析
- 软件端编写的代码, 例如`int a=2`, 是直接映射, a的地址直接对应着数据2, 而[[202605171319-指针|指针]]则是间接映射, 其地址下映射着另一个地址, 而这个地址才有着值...
- 所以, 任意数据结构, 都可能有键, 例如[[202605232106-链表|链表]], 有无数个间接映射, 而[[202605171319-指针|指针]], 实际上也是映射的一种表示, 由此拓展出来, →, 似乎就是映射, 有→, 似乎一定能通过
